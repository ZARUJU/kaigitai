# リポジトリガイドライン

## 開発の方法

単なるfix等では行わないけど

1. ユーザーからのタスク依頼
2. `docs/task/XXX_hogehoge.md`に実施内容を整理
   - `docs/task` の書き方（見出しと箇条書きを利用のこと）: 
     - h1 GOAL
      - {ゴールA}
      - {ゴールB}
      - h1 ユーザーからの依頼
      - h1 タスク
      - h2 現状
      - h2 変更後
      - h2 手順
3. ユーザーと話し合い、内容をブラッシュアップ
4. ユーザーの同意に基づき実装開始
5. 実装後、コミット
   - この際、コミットメッセージは Conventional Commits に従う文面を提案すること
   - `feat:`など以外の詳細部分は日本語で提案のこと

## 本ソフトについて

- 散逸しているさまざまな団体の会議データを集約したデータベース
- 細かい仕様は`docs/spec/XXX_*.md`に記載
- まず、`docs/spec/001_overview.md`で概要を把握のこと
